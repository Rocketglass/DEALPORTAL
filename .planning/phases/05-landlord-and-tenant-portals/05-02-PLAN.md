  <action>
Create `src/lib/queries/landlord.ts` with three query functions that use RLS-filtered Supabase queries (the logged-in landlord's session automatically filters data via RLS policies from Phase 3).

**getLandlordProperties(contactId: string)**
- Query `properties` table where `owner_contact_id = contactId`
- Join units, and for each property count applications (via `applications` table where `property_id` matches)
- Also join LOIs and leases counts per property for pipeline display
- Return type: `LandlordProperty[]` with fields: property info + unitCount + applicationCount + activeLoiCount + activeLeaseCount
- If user is an agent (has principalId), use principalId as the contactId (agent acts on behalf of landlord)

**getLandlordApplications(contactId: string)**
- Query `applications` table joined with `properties` where `properties.owner_contact_id = contactId`
- Include contact (tenant info), property, documents relations
- Order by created_at desc
- Return type: `ApplicationWithRelations[]`

**getLandlordApplication(applicationId: string, contactId: string)**
- Query single application by ID with full relations (property, unit, contact, documents)
- Verify the application's property belongs to this landlord (owner_contact_id = contactId)
- Return type: `ApplicationWithRelations | null`

All three functions follow the `{ data, error }` return pattern used in all other query files. Import createClient from `@/lib/supabase/server`.

Also export a helper `getEffectiveContactId(user: AuthUser): string` that returns `user.principalId ?? user.contactId` -- this handles the agent delegation pattern (agents use their principal's contactId).
