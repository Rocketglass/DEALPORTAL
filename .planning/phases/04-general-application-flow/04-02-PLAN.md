The Properties step:
- On mount, fetch `GET /api/public/properties` to get the list of available properties
- Display properties as selectable cards in a responsive grid (2 columns on mobile, 3 on desktop)
- Each card shows: property name, address (city, state), property type
- Cards have a selected state (blue border + checkmark) — clicking toggles selection
- Store selected property IDs in state: `selectedPropertyIds: string[]`
- Read `?property=UUID` query param on mount — if present, pre-select that property
- Validation: at least one property must be selected (show error "Please select at least one property you're interested in")
- Include a "Not sure yet — show me all options" option that selects all properties (or allows skipping)

**2. Auth-aware draft persistence:**
- On mount, check auth state via Supabase browser client `supabase.auth.getUser()`
- Store auth state in `const [authUser, setAuthUser] = useState<{ id: string; email: string } | null>(null)`
- If authenticated AND server draft exists (GET /api/applications/drafts returns non-null):
  - Merge server draft into form state (server draft takes priority over localStorage)
  - Restore selectedPropertyIds and currentStep from server draft
  - Show a toast/banner: "Welcome back! Your application draft has been restored."
- If not authenticated: continue using localStorage only (existing behavior)
- On each step advancement (goNext), if authenticated: fire PUT /api/applications/drafts in background (fire-and-forget, non-blocking)
- Keep localStorage persistence for ALL users (including authenticated) as a fallback

**3. Auth-gated submission (Step 6 - Review):**
- At the Review step, check if user is authenticated
- If NOT authenticated: instead of the submit button, show a prominent card:
  ```
  "To submit your application, please create an account or log in."
  [Create Account] [Log In]
  ```
  - "Create Account" links to `/register?redirect=/apply&draft=restore`
  - "Log In" links to `/login?redirect=/apply&draft=restore`
  - After redirect back, the form restores from localStorage (already works) and server draft sync kicks in
- If authenticated: show the submit button as before
- On submit: include `propertyIds: selectedPropertyIds` in the POST body to `/api/applications`
- After successful submit: also clear the server draft via `PUT /api/applications/drafts` with empty data (or simply let it be — the draft will be stale but harmless). Actually, just clear localStorage. Server draft cleanup is optional.

**4. Query param handling for QR code flow:**
- Read `searchParams.get('property')` on mount
- If a property UUID is present, pre-select it in the multi-property selector
- Do NOT lock the form to that property — user can deselect it or add others

**5. Preserve all existing form logic:**
- Keep all field validation, phone formatting, currency formatting, file upload with drag-drop
- Keep the progress stepper UI
- Keep the existing mobile-first responsive layout
- Update step number references throughout (everything shifts +1 after step 1)
- Update STORAGE_KEY to `'rr_application_draft_v2'` to avoid conflicts with old drafts

- Loading state for properties fetch: skeleton cards (3 pulsing gray rectangles)
- Mobile: property cards stack to single column below sm breakpoint
